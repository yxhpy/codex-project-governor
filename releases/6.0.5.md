# 6.0.5 - Session Learning Memory Loop

## Added

- Added `.project-governor/state/COMMAND_LEARNINGS.json` for one-off failed commands, error signatures, and corrective lessons.
- Added `.project-governor/state/MEMORY_HYGIENE.json` for stale, superseded, or bloated memory candidates that need review before pruning or marking superseded.
- Added `skills/memory-compact/scripts/record_session_learning.py` to classify session learnings and write them to the right layer.
- Added tests proving command failures, repeated mistakes, stale memory candidates, and secret-like items are handled correctly and are retrievable through memory search.

## Workflow

- At non-trivial session start, run memory search for prior command failures, repeated mistakes, stale-memory notes, decisions, and task history related to the request.
- Before final response, if a command failed, an assumption was corrected, or memory looked stale, run:
  `python3 skills/memory-compact/scripts/record_session_learning.py --project . --input <json> --apply`
- One-off failures stay in `COMMAND_LEARNINGS.json`; repeated mistakes are promoted to `docs/memory/REPEATED_AGENT_MISTAKES.md`; stale memories go to `MEMORY_HYGIENE.json`.

## Compatibility

- No new third-party Python dependencies.
- The new ledgers are project-owned `.project-governor/state/` files and are indexed by `context-indexer --memory-search`.
- Existing initialized projects should add the new state files through `plugin-upgrade-migrator`; local `AGENTS.md` edits are not overwritten.
